# 要件定義（Requirements）

## 目的

MiraStudy（ミラスタディ）は、小中学生向けのAI学習支援アプリケーションです。
文部科学省の学習指導要領に基づき、個別最適化された学習体験を提供します。

## スコープ

### 対象（本リポジトリで扱う）

- 小学生・中学生向けの学習コンテンツ生成
- 文科省公開PDFの自動取得・検証機能
- プロファイル管理（2ユーザーと保護者ロール）
- 学年・教科に応じた問題出題とフィードバック
- 保護者向けダッシュボード（データソース管理）
- Reactフロントエンド + Python FastAPIバックエンド
- Vercelへのデプロイ

### 非対象（本リポジトリでは行わない）

- 高校生以上の学習コンテンツ
- ユーザー認証・課金機能（MVP段階）
- モバイルネイティブアプリ（WebアプリのみPWA
- 学習履歴の永続化（将来フェーズで検討）
- リアルタイムAI解析（初期はモック応答）

## 用語

| 用語 | 定義 |
|---|---|
| 正本（SSOT） | Single Source of Truth。docs/ 配下の文書を唯一の信頼源とする |
| 受入条件（AC） | Issue/PR が完了と判断できる基準 |
| MEXT | 文部科学省（Ministry of Education, Culture, Sports, Science and Technology） |
| Provenance | データの出所・実在性を追跡可能にする仕組み |
| 学年判定ロジック | 生年月日から現在の学年を4月1日基準で自動計算する |

## ユーザープロファイル

| プロファイル | ID | ダミー生年月日 | 2026年3月時点の年齢・学年 |
|---|---|---|---|
| ユーザーA | user_a | 2010-05-15（ダミー） | 15歳・中学校3年生 |
| ユーザーB | user_b | 2011-08-20（ダミー） | 14歳・中学校2年生 |

> **注意**: 実名・実生年月日はコード・ドキュメントに記載しない（個人情報保護）。上記はダミーデータ。

## 成功条件（ゴール）

- ユーザーは学年に応じた教科を選択し、単元を入力できる
- 文科省PDFの実在URLが自動取得・検証され、保護者ダッシュボードで確認できる
- 学習問題が生成され、解答・フィードバックが表示される
- Vercelにデプロイされ、ブラウザからアクセス可能
- CI が品質ゲートとして機能し、失敗時はマージできない
- フロントエンド・バックエンドが統合され、エンドツーエンドで動作する

## 機能要件（FR）

### FR-001 プロファイル選択

- 入力：ユーザーがプロファイル（ユーザーA or ユーザーB or 保護者）を選択
- 出力：選択されたプロファイルに応じたホーム画面を表示
- ロジック：生年月日から学年を自動計算（4月1日基準）
- 担当モジュール：frontend/App.jsx（初期）、後に components/Login.jsx

### FR-002 教科選択

- 入力：ユーザーの学年に応じた教科一覧を表示
- 出力：選択された教科の入力画面へ遷移
- ルール：小学生は「算数・国語・理科・社会・外国語・生活・図画工作・体育」、中学生は「数学・国語・理科・社会・英語・保健体育・美術・技術家庭」
- 担当モジュール：frontend/components/SubjectSelector.jsx

### FR-003 単元入力

- 入力：学習する単元名をテキスト入力（例：「連立方程式」「小数のかけ算」）
- 出力：AI解析画面へ遷移
- バリデーション：空文字は送信不可
- 担当モジュール：frontend/components/UnitInput.jsx

### FR-004 MEXT PDF自動取得・検証（保護者機能）

- 入力：保護者が「最新URLを取得」ボタンをクリック
- 処理：
  1. 文科省Webサイトへリクエスト送信
  2. 最新のPDF配布パスを自動特定
  3. URLの実在性を確認 (HTTP 200 OK)
  4. 学年への適合性を検証
  5. ナレッジベースへ同期
- 出力：検証済みURLリストを表示（リアルタイムステップ表示）
- 担当モジュール：backend/services/mext_fetcher.py

### FR-005 問題生成・解答・フィードバック

- 入力：単元名、学年、教科
- 処理：AIが文科省データをもとに問題を生成
- 出力：問題文を表示し、ユーザー解答を受付
- フィードバック：正解/不正解を判定し、解説を表示- 担当モジュール：backend/services/question_generator.py + frontend/components/Quiz.jsx

### FR-006 モード切替（通常学習 / テスト復習）

- 入力：ユーザーがモード選択ボタンをクリック
- 出力：選択されたモードに応じてUIカラーとラベルを変更
- 担当モジュール：frontend/App.jsx（State管理）

## 非機能要件（NFR）

### NFR-001 再現性

- フロントエンド：package.jsonでバージョン固定
- バックエンド：pyproject.tomlまたはrequirements.txtでバージョン固定
- 同一入力に対して同一出力が得られることをテストで担保する

### NFR-010 セキュリティ

- 秘密情報（APIキー、トークン）をコミットしない
- 環境変数（.env）で管理し、Vercelのシークレット機能を使用
- CI の policy_check で機械的に検査する

### NFR-020 保守性

- コンポーネントは責務ごとに分割する
- 変更は 1PR で理解可能な粒度に分割する
- コードの識別子は英語、説明文・PR・Issue は日本語で記述する
- Reactコンポーネントは1ファイル500行以下を目安とする

### NFR-030 運用容易性

- ローカルで最小コマンドで実行できる（`npm start`, `uvicorn main:app`）
- 失敗時の復旧手順を runbook に記載する
- Vercelデプロイはmainブランチへのpush時に自動実行

### NFR-040 パフォーマンス

- 初回ロード時間: 3秒以内（Lighthouse Score 80以上目標）
- PDF取得処理: 10秒以内にレスポンス
- 問題生成: 5秒以内に表示開始

### NFR-050 アクセシビリティ

- カラーコントラスト比 4.5:1 以上
- キーボード操作のみで全機能利用可能
- セマンティックHTML使用（ボタンは`<button>`、リンクは`<a>`）

## 受入条件（全PR共通）

| ID     | 条件                                                                    |
| ------ | ----------------------------------------------------------------------- |
| AC-001 | 変更は要件・ポリシー・計画のいずれかに整合する                          |
| AC-010 | 必要なテストが追加または更新されている                                  |
| AC-020 | CI（lint/type/test/policy_check）が成功している                         |
| AC-030 | 変更に応じて正本（docs）が更新されている                                |
| AC-040 | 検証手順と結果が PR 本文に記載されている                                |
| AC-050 | プロジェクト固有の制約に反する変更がない（例外は ADR に記載されている） |
| AC-060 | フロントエンドの変更はローカルで `npm start` で動作確認済み             |
| AC-070 | バックエンドの変更はローカルで `pytest` が成功している                  |
| AC-080 | UIの変更はスクリーンショットまたは動画をPRに添付する                    |
